# 제작현황(ProductionListMain) 성능 리팩토링 지침

`src/components/production/ProductionListMain.tsx` (약 3,700줄) — 제작현황 탭의 PCB/Cable 테이블 인라인 편집이 셀 클릭 시 느리게 반응하는 문제를 조사한 내용과, 향후 리팩토링 방향을 정리한 문서.

## 현재 증상

- 테이블 셀을 클릭(선택) → 다시 클릭(편집 진입)할 때 즉각 반응하지 않고 약 0.1~0.3초 지연이 느껴짐.
- 데이터는 많지 않음 (PCB 44건 + Cable 31건 = 총 75행, 2026-07-04 기준 실측).

## 근본 원인

### 1. 셀 하나가 바뀌어도 표 전체를 다시 계산함 (핵심 원인)
`ProductionListMain`은 3,700줄짜리 단일 컴포넌트로, `React.memo`/`useMemo`/`useCallback`이 전혀 적용되어 있지 않다. `selectedCells`/`editingCell` 같은 state가 하나만 바뀌어도 컴포넌트 전체가 다시 실행되며, 행마다 `renderEditableCell`이 20~30번 호출된다(테이블 전체 소스에서 48회 호출 지점 확인). 75행 × 약 30컬럼 = 약 2,250개 셀이 클릭 한 번마다 전부 다시 계산된다.

이때 비용이 드는 건 실제 DOM 반영(이미 최소한만 반영됨)이 아니라, React가 "무엇이 바뀌었는지" 비교하기 위해 2,250개 셀의 설명(React element)을 매번 새로 만들어내는 JS 계산 비용이다. 셀 하나당 `parseColorState` 문자열 파싱, className 템플릿 리터럴 조립, 매 렌더마다 새로 생성되는 style 객체, `selectedCells.includes()` 등이 실행된다.

실측(2026-07-04, 브라우저 클릭 → DOM 갱신까지 MutationObserver로 측정): **150~360ms**.

### 2. 전체 데이터를 항상 다 그림 (스크롤 밖 포함) — "가상화" 부재
Excel/Google Sheets 같은 스프레드시트는 시트가 100만 행이어도 화면에 보이는 30~40행만 그린다(가상화/윈도잉). 이 표는 그 최적화가 없어서, 화면에 보이지 않는 행까지 포함해 항상 전체를 렌더링한다. **데이터가 쌓일수록 클릭 반응이 선형으로 느려지는 구조** — 지금은 75행이라 체감이 크지 않지만, 500행이 쌓이면 이론상 6~7배 느려진다.

### 3. dev 모드 오버헤드 (부차적)
측정은 Vite dev 서버(개발용 React 빌드) 기준. 실제 배포 빌드는 압축·최적화되어 있어 통상 2~5배 빠르다. 사용자가 실사용 중 체감하는 지연이 자동화 측정치(150~360ms)보다 작게 느껴지는 이유 중 하나.

## 이번에 이미 고친 것

1. **셀 클릭 → 선택 → 재클릭 → 편집 진입** 흐름 구현 (`handleCellClick`, `handleCellMouseDown`을 ref 기반으로 변경 — mousedown 시점에 곧바로 `selectedCells`를 채우면 뒤이은 click이 "이미 선택됨"으로 오판해 1클릭에 바로 편집 진입하는 버그가 있었음).
2. 선택된 셀의 `transition-colors` CSS를 제거해 하이라이트가 150ms 페이드 없이 즉시 나타나도록 수정.
3. `AppLayout.tsx`에서 데스크톱/모바일 뷰가 `<AppRoutes />`를 각각 별도로 마운트해 **앱 전체 콘텐츠가 상시 2벌 렌더링**되던 문제 발견 및 수정 (CSS 변수 기반 반응형 margin으로 단일 마운트화). 클릭 반응 속도 자체에는 영향 없었지만(각 마운트가 독립된 state를 가져 클릭 시 보이는 쪽만 리렌더됨), 초기 로딩 비용·메모리·Supabase Realtime 구독 중복 등 앱 전체 낭비를 절반으로 줄임.

## ✅ 근본 개선 완료 (2026-07-05, 아래 방향대로 구현됨)

두 가지 모두 구현 완료:

1. **가상화**: 각 테이블을 세로 스크롤 컨테이너(`maxHeight: calc(100vh - 220px)`)로 감싸고,
   스크롤 위치 기준 보이는 행 + 버퍼(12행)만 렌더. 전체 높이는 tbody 스페이서 `<tr>`로 유지.
   행 높이는 마운트 후 실측 보정. 헤더 2행은 CSS sticky(top 0 / 15px)로 컨테이너 상단 고정,
   좌측 고정 '코너' 헤더는 인라인 zIndex 40. (rAF 스로틀은 백그라운드 탭에서 콜백이 보류되는
   함정이 있어 스크롤마다 직접 재계산 — setState는 값이 같으면 재렌더 생략)
2. **행 단위 렌더 격리**: 행 렌더 본문을 `renderPcbRow`/`renderCableRow`로 추출하고 `MemoRow`
   (React.memo + 커스텀 비교: item 참조·index·rowSig·칼럼폭 캐시)로 감쌈. 행과 관련된 UI 상태
   (선택 셀·편집 중 셀·색상 피커)는 `rowSig` 문자열로 요약해 바뀐 행만 재렌더.
   행 내부 커스텀 이벤트 핸들러 9종은 `useStableHandler`(동일 함수 객체 + 최신 로직 실행)로
   안정화해 렌더 스킵된 행의 stale closure 문제를 차단.
3. **칼럼폭 실측 캐시**: 렌더마다 (호출 지점 수 × 전체 행) canvas 실측이 돌던 `getColumnWidth`를
   테이블별 `useMemo` 일괄 계산(`pcbColumnWidths`/`cableColumnWidths`)으로 교체.

실측(2026-07-05, dev 모드, PCB 977행 전체 보기):
- DOM에는 항상 ~60행만 존재. 스크롤 시 해당 구간으로 즉시 교체(끝 977행 도달 확인).
- 셀 클릭 → DOM 반영: **가상화만 55~313ms → 행 격리 후 50~67ms** (975행 기준. 배포 빌드는 통상 2~5배 빠름).
- 편집 진입/ESC, 색상 피커, 드래그 다중선택 회귀 없음.

## (참고) 당초 남은 근본 개선 방향 (위로 구현 완료)

Excel과 같은 원리(다른 구현)로 데이터 양과 무관하게 항상 빠른 반응속도를 유지하려면 두 가지가 필요:

1. **가상화(virtualization)**: 화면에 보이는 행만 렌더링 (예: `react-window`, `@tanstack/react-virtual`). 데이터가 얼마나 쌓이든 항상 "화면에 보이는 분량"만큼만 계산하게 됨. 초기 로딩·스크롤·클릭 반응 모두에 적용됨.
2. **셀 단위 메모이제이션**: 각 셀을 독립된 컴포넌트로 분리하고 `React.memo`로 감싸, 자신의 데이터가 안 바뀌면 리렌더를 건너뛰게 함. 클릭 반응이 전체 데이터 양과 무관해짐(행 단위보다 더 정확한 목표 — 행 단위는 클릭한 행의 나머지 칸까지 다시 계산하지만, 셀 단위는 실제 바뀐 1~2칸만 계산).

### 비용/위험
- 3,700줄 단일 파일 안에 드래그 다중선택, 색상 피커, ARTWORK 하이브리드 에디터, 입고대기 버튼, 벤더 자동완성 등이 모두 얽혀 있어 회귀 위험이 있는 작업.
- 드래그 다중선택 기능이 "선택 범위"를 표 전체 단위로 계산해야 해서, 상태를 부모 컴포넌트에 모아두는 현재 구조와 맞물려 있음 — 셀 단위로 쪼갤 때 이 부분을 함께 고려해야 함.

### 진행 판단 기준
- 지금 체감 지연(0.1~0.2초)은 이미 쓸 만한 수준이라 당장 막히는 문제는 아님.
- 리팩토링의 가치는 "오늘 더 빠르게"가 아니라 "제작 이력이 계속 쌓여도 반응속도가 나빠지지 않게 미리 방지"하는 데 있음.
- 월별 필터 기준으로 몇십~백 단위 규모가 안정적으로 유지될 전망이면 당장 급하지 않고, 데이터가 계속 누적될 시스템이면 선제적으로 진행하는 게 맞음.
